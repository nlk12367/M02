什麼是大語言模型

  • 簡單來說 LLM 就是一種 電子大腦 ，為什麼呢?
    因為其構造就是以人類大腦的細微構造及傳輸方式作為雛型；在模型中運算資源的結點叫 hypothesis 激發函數
    也就是人類的神經元，詳細介紹請參考 圖(1):



    圖(1)
    所以，學界也稱其為 類神經網路

  • 另外，在人類的世界中常常會遇到要分析事件的狀況， 最常見的方法是撰寫計畫書。
    透過撰寫計畫書可以獲得 三大優點，以下將仔細說明:

    • 明確化: 使人們能夠將抽象的想法轉化成具體可實現之方案。
    • 高效率: 減少資訊不對稱性，讓溝通與執行更順暢。
    • 提升成功率: 事先進行全面的規劃，提升專案成功的可行性。

    了解撰寫計劃書的效益，有助於我們理解如何與 LLM 溝通~

  • 基於上述，聰明的你肯定能發現與 LLM 良好溝通的訣竅就是撰寫計劃書(沒發現的話，就多看幾次內文 xd)；在 AI的世界觀中，他們的計畫書叫 Prompt Engineering 中文是 提示詞工程，在下一章中我將向您解密 提示詞工程。

Prompt−EngineeringPrompt-EngineeringPrompt−Engineering

  • 在這個 AI 普及的世代，大家的差距已經不是在比誰的電腦性能好，變成在比誰能 下對指令 讓 AI 跑出更優的成果，而 Prompt Engineering 就是拉開彼此距離的關鍵。

  • 撰寫提示詞的標記語法有很多，常見的有 AsciiDoc、Markdown、reStructuredText 三種，以下將說明特色與用處:

    • AsciiDoc : 具標準化、多種原生功能 等特色，常用於建置 XML 文檔，但運用生態較小。
    • Markdown : 是三種語法中最簡單的語法、運用生態較廣，常用於撰寫筆記與文章，另外還能嵌入 HTML 使文章更具多樣性，但原生不支援表格、目錄、以及數學符號 等標記功能。
    • reStructuredText : 具標準化、擴充性高 等特色，擁有專職工具 Sphinx 進行內容轉換建立文件，常用於撰寫涵高度程式碼的說明文件(EX: Github)，但規則較生硬。

      以上內容參考 Lightweight Markup: AsciiDoc, reStructuredText, and Markdown Compared

    雖然 AsciiDoc 與 reStructuredText 功能都非常強大，但受運用生態圈限制 以及 AI 訓練文本類型數量影響，多數 Prompt Engineering 都採用 Markdown 來寫 Prompt。

  • 目前大家所了解的 Markdown 是採用混和型態的語法，主要由 AsciiDoc + HTML + XML + Markdown 所構成，以下將為您說明 Markdown 語法的規則

    Markdown Most Common Syntax :

      '#' - Heading
  **text** - Bold
  *text* - Italic
  [text](url) - Link
  ![alt](url) - Image
  - item - List
  > quote - Blockquote
  `code` - Code

    Memory Tricks :

      # Hash looks like stairs, indicates heading level
  * Asterisk means emphasis, one for italic, two for bold
  []() Square brackets for text, parentheses for link
  ![]() Exclamation mark for image
  > Greater-than looks like an arrow, points to quote

    以上內容取材自 Why Use Markdown?

  • 這裡提供給各位讀者幾個 prompt 的設計範例:

    • 單次動作輸出

    # 任務
    -解析文章
    # 任務說明
    - step1.**完整讀取**附件內容
    - step2,以不篡改文章原意為前提,向我詳細說明這篇論文的內容
    # 限制
    - 呈現內容之來源只能來自附件文章內容



    • 長文本型輸出

    # 任務
    - 分析與總結內容

    # 任務說明
    - 輸出要有標題，結語要根據使用者需求去決定，並且生成的內容針對文章本身，去除網頁的廣告內容。 - 要先完整分析全文上下文 - 資料來自數據庫 - 不要用markdown語法輸出，以json格式輸出。 - 文章最後要設定"P":"1"

    # 資訊過濾與遞補
    - 假設文章0 的 "title"、"text"、"link"其中一個不存在，則將"文章1"的資訊自動遞補"文章0" - 輸出還是只能輸出5個份文章，編號從'0'開始數。

    # 限制
    - 若有文章出現只有標題 沒有內文，那麼這篇文章直接略過，不進行分析 - 以json格式輸出 - 每篇文章的重點整理上限為300字

    # 範例輸出格式
    "文章0標題": "原文標題",
    "文章0重點整理"："你整理的內容" ,
    "文章0時間": "原文時間",
    "文章0連結": "原文連結",
    "文章1標題": "原文標題",
    "文章1重點整理"："你整理的內容" ,
    "文章1時間": "原文時間",
    "文章1連結": "原文連結",
    "P":"1"

使用心得

  • 之前 有測試過 Opus 4.5 與 Grok-4 在寫程式碼 與 debug 的功能，實驗結果表現良好，邏輯錯誤率較 GPT-5.1 少很多；
  • 如果您使用是 GPT 生成學習講義相關的任務 ，可以在 Prompt 末端加上
    +++
    # 限制
    - 進行生成任務時，只能參考 數據庫 的內容
    +++
    這樣提示可以降低 GPT 出現幻覺機率，提高您生成內容的品質。 (以往 GPT出現幻覺的機率偏高)
