## **什麼是大語言模型** 
-   簡單來說 LLM 就是一種 **電子大腦** ，為什麼呢? <br>因為其構造就是以**人類大腦**的細微構造及傳輸方式作為雛型；在模型中運算資源的結點叫 **hypothesis** **激發函數** <br> 也就是人類的神經元，詳細介紹請參考 **圖(1)**:<br><br><br>

-   另外，在人類的世界中常常會遇到要分析事件的狀況， 最常見的方法是撰寫計畫書。<br> 透過撰寫計畫書可以獲得 **三大優點**，以下將仔細說明:<br><br>
    * **明確化**: 使人們能夠將抽象的想法轉化成具體可實現之方案。  
    * **高效率**: 減少資訊不對稱性，讓溝通與執行更順暢。<br> 
    * **提升成功率**: 事先進行全面的規劃，提升專案成功的可行性。<br><br>

    了解撰寫計劃書的效益，有助於我們理解如何與 LLM 溝通~

## **$$Prompt-Engineering$$**

- 在這個 AI 普及的世代，大家的差距已經不是在比誰的電腦性能好，變成在比誰能 **下對指令** 讓 AI 跑出更優的成果，而 **Prompt Engineering** 就是拉開彼此距離的關鍵。<br>
- 撰寫提示詞的標記語法有很多，常見的有 **AsciiDoc**、**Markdown**、**reStructuredText** 三種，以下將說明特色與用處:<br><br>
    *   **AsciiDoc** : 具標準化、多種原生功能 等特色，常用於建置 XML 文檔，但運用生態較小。<br>
    *   **Markdown** : 是三種語法中最簡單的語法、運用生態較廣，常用於撰寫筆記與文章，另外還能嵌入 HTML 使文章更具多樣性，但原生不支援表格、目錄、以及數學符號 等標記功能。<br>
    *   **reStructuredText** : 具標準化、擴充性高 等特色，擁有專職工具 **Sphinx** 進行內容轉換建立文件，常用於撰寫涵高度程式碼的說明文件(EX: Github)，但規則較生硬。<br>
    <br>以上內容參考 [Lightweight Markup: AsciiDoc, reStructuredText, and Markdown Compared](https://readmedium.com/lightweight-markup-asciidoc-restructuredtext-and-markdown-compared-d5a33fedcac9)<br><br>

    雖然 AsciiDoc 與 reStructuredText 功能都非常強大，但受運用生態圈限制 以及 AI 訓練文本類型數量影響，多數 Prompt Engineering 都採用 Markdown 來寫 Prompt。

    
- 目前大家所了解的 Markdown 是採用混和型態的語法，主要由 AsciiDoc + HTML + XML + Markdown 所構成，以下將為您說明 Markdown 語法的規則 <br><br>
    > Markdown Most Common Syntax :

        '#' - Heading
        **text** - Bold
        *text* - Italic
        [text](url) - Link
        ![alt](url) - Image
        - item - List
        > quote - Blockquote
        `code` - Code

    > Memory Tricks :

        # Hash looks like stairs, indicates heading level
        * Asterisk means emphasis, one for italic, two for bold
        []() Square brackets for text, parentheses for link
        ![]() Exclamation mark for image
        > Greater-than looks like an arrow, points to quote

    以上內容取材自 [Why Use Markdown?](https://www.markdownlang.com/intro/why-use-markdown.html) 

- 這裡提供給各位讀者幾個 prompt 的設計範例:<br>

    * 單次動作輸出
    ---- 

    <br>
    # 任務<br>
        -解析文章<br>
    # 任務說明<br>
        - step1.**完整讀取**附件內容<br>
        - step2,以不篡改文章原意為前提,向我詳細說明這篇論文的內容<br>
    # 限制<br>
        - 呈現內容之來源只能來自附件文章內容<br>

    ---- 
    <br><br>
    * 人機互動型輸出
    ---- 

    <br>
     # 任務<br>
        - 設計一位名為**「效率軌跡」**的**「個人行為經濟學策劃師」**，目標是協助用戶**建立最優化的行為消費曲線**。 
        <br><br>
     # 任務說明<br>
        - 他的對話模式必須模擬APP**交易與獎勵循環**的架，每一輪互動皆必須固定遵循階段確認、評估、行動指引的三步驟流程線。語氣必須**充滿實用主義的積極肯定**，並堅持將用戶的**每項進步轉化為可兌換的『星級獎勵**，以完成行為編程。

## **各模型差異**
- 在前面的章節我們已經摸清了 LLM 的底層邏輯，而這部分我們會以 **基準測試** 呈現各 LLM 能力差異， 並分享使用心得，讓各位在使用 LLM 時能更得心應手。<br><br>

    * OpenAI-GPT-5.1 :

        *   Artificial Analysis 智慧指數 : 70
        *   Speed 每秒生成量 : 152.2 Token/s 
        *   Verbosity 生成冗長度 81M Tokens
        *   特長 : 日常陪伴、社群文案創作
    <br>
    * Google-Gemini-3-Pro 

        *   Artificial Analysis 智慧指數 : 73
        *   Speed 每秒生成量 : 138.9 Token/s 
        *   Verbosity 生成冗長度 92M Tokens
        *   特長 : 科學研究、複雜邏輯運算、龐大且完整原生生態圈 (GCP)
    <br>
    * xAI-Grok-4

### 使用心得
- 之前 有測試過 Opus 4.5 與 Grok-4 在寫程式碼 與 debug 的功能，實驗結果表現良好，邏輯錯誤率較 GPT-5.1 少很多； 
- 如果您使用是 GPT 生成學習講義相關的任務 ，可以在 Prompt 末端加上<br>+++<br># 限制 <br>- 進行生成任務時，只能參考 **數據庫** 的內容<br>+++<br>這樣提示可以降低 GPT 出現幻覺機率，提高您生成內容的品質。 (以往 GPT出現幻覺的機率偏高)
